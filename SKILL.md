# Dev Blog Writer Skill — 2026 Global Standard

## 목적
실무 경험과 기술적 깊이를 보여주는 개발 블로그 글을 작성한다.
2026년 SEO + E-E-A-T + LLM 청킹 최적화 기준 적용.

## 핵심 원칙

### E-E-A-T (Experience, Expertise, Authoritativeness, Trust)
- **Experience**: 직접 겪은 실무 경험 기반 (이론만 나열 금지)
- **Expertise**: 깊이 있는 기술 분석과 코드 예시
- **Authoritativeness**: 벤치마크, 공식 문서 인용, 정량 근거
- **Trust**: 한계점과 트레이드오프 솔직하게 명시

### Hub-and-Spoke 콘텐츠 전략
- 하나의 핵심 주제(Hub)에 대해 깊이 있는 글을 쓰고
- 관련 세부 주제(Spoke)로 연결하는 시리즈 구성
- 수십 개의 얕은 포스트보다 잘 연결된 클러스터 1개가 효과적

### LLM 친화적 구조
- 명확한 Heading 계층 (H1 > H2 > H3)
- 짧은 문단 (3~5문장)
- 코드 블록에 언어 명시
- 핵심 내용을 각 섹션 첫 문장에 배치

## 블로그 글 유형별 템플릿

### Type 1: 기술 심화 (Deep Dive)
실무에서 특정 기술을 깊이 파고든 경험을 공유.

```markdown
# [기술명]: [구체적 주제] — [핵심 인사이트 한 줄]

> **TL;DR**: 이 글의 핵심을 2~3문장으로. 바쁜 독자를 위한 요약.

## 왜 이 글을 쓰는가
- 어떤 문제를 만났는가
- 기존 자료에서 부족했던 점
- 이 글에서 다루는 범위

## 배경 지식 (선택)
- 이 글을 이해하기 위한 최소 사전 지식
- 핵심 개념 한 줄 정의 (처음 등장하는 용어)

## 핵심 내용

### [소주제 1]
**문제**: ...
**분석**: ...

\```python
# 실무 코드 예시 — Before
# (문제가 있는 코드)
\```

\```python
# 개선된 코드 — After
# (해결된 코드 + 주석으로 핵심 포인트)
\```

**왜 이렇게 했는가**: 기술 선택의 이유와 대안 비교

### [소주제 2]
(동일 구조)

## 벤치마크 / 측정 결과 (해당 시)
| 항목 | Before | After | 개선율 |
|------|--------|-------|--------|
| 응답시간 (p95) | 450ms | 45ms | 90% ↓ |
| 메모리 사용 | 2.1GB | 800MB | 62% ↓ |

## 주의할 점 / 함정
- 이 접근법의 한계
- "이렇게 하면 안 된다" 안티패턴
- 적용 가능한 상황 vs 불가능한 상황

## 정리
- 핵심 포인트 3가지 bullet
- 다음에 읽으면 좋은 글 (시리즈 연결)

## References
- [공식 문서](url)
- [관련 논문/블로그](url)

---
*이 글에서 사용한 환경: Python 3.12, FastAPI 0.115, PostgreSQL 16*
*작성일: YYYY-MM-DD · 최종 수정: YYYY-MM-DD*
```

### Type 2: 트러블슈팅 스토리 (Debugging Story)
실무에서 겪은 버그/장애를 해결한 경험 공유. 면접관이 가장 좋아하는 유형.

```markdown
# [에러/현상]: [근본 원인을 한 줄로]

> **TL;DR**: [문제] → [원인] → [해결] 한 줄 요약

## 증상
- 언제, 어디서, 어떤 조건에서 발생했는가
- 에러 메시지 또는 이상 동작 (코드/로그 첨부)
- 영향 범위 (사용자 수, 서비스 영역)

## 초기 가설과 삽질
### 가설 1: [첫 번째 추측]
- 왜 이렇게 생각했는가
- 검증 방법과 결과
- **결과**: ❌ 원인이 아니었다 (왜?)

### 가설 2: [두 번째 추측]
- (동일 구조)

## 근본 원인 (Root Cause)
- 5 Whys 또는 원인 분석 체인
- 코드 레벨에서의 정확한 원인 지점

\```python
# 문제의 코드 (라인 하이라이트)
\```

## 해결
\```python
# 수정된 코드
\```

- 왜 이 수정이 올바른가
- 사이드 이펙트는 없는가

## 재발 방지
- [ ] 테스트 추가 (어떤 케이스)
- [ ] 모니터링/알림 설정
- [ ] 코드 리뷰 체크리스트 항목 추가
- [ ] 문서화

## 교훈
- 이 경험에서 일반화할 수 있는 원칙
- "앞으로 비슷한 상황에서는..."

---
*발생 환경: ...*
*해결 소요 시간: ...*
```

### Type 3: 비교 분석 (Comparison / Decision)
기술 선택의 과정과 근거를 공유. 의사결정 능력을 보여줌.

```markdown
# [기술A] vs [기술B]: [사용 맥락에서의 선택 기준]

> **TL;DR**: [상황]에서는 [선택]이 적합. 이유: [핵심 근거 1개]

## 비교 배경
- 어떤 상황에서 이 선택이 필요했는가
- 평가 기준 (성능, DX, 생태계, 비용, 학습 곡선)

## 비교 요약
| 기준 | [기술A] | [기술B] | 승자 |
|------|---------|---------|------|
| 성능 | ... | ... | |
| DX | ... | ... | |
| 생태계 | ... | ... | |
| 비용 | ... | ... | |
| 학습 곡선 | ... | ... | |

## 상세 비교

### [기준 1]: 성능
(벤치마크 코드 + 결과)

### [기준 2]: 개발자 경험
(코드 예시 비교)

## 결론: 우리는 [X]를 선택했다
- 선택 이유
- 선택하지 않은 기술이 더 적합한 상황
- 6개월 후 회고 (선택이 옳았는가?)
```

### Type 4: 튜토리얼 / How-to
실무에서 검증된 구현 방법을 단계별로 안내.

```markdown
# [목표]: [기술 스택으로] 단계별 구현 가이드

> **TL;DR**: [무엇]을 [어떻게] 구현. 완성 코드 → [GitHub 링크]

## Prerequisites
- 필요한 사전 지식 / 설치 환경
- 최종 결과물 미리보기 (스크린샷/GIF)

## Step 1: [단계명]
설명 + 코드

## Step 2: [단계명]
설명 + 코드

## Step N: [최종 단계]

## 실무 적용 시 주의점
- 튜토리얼과 프로덕션의 차이
- 보안, 에러 처리, 스케일링 고려사항

## 전체 코드
[GitHub 저장소 링크]
```

### Type 5: 회고 / 아키텍처 결정 기록 (ADR Blog)
프로젝트 설계 결정을 공유. 시니어 엔지니어 관점 어필.

```markdown
# [프로젝트명] 아키텍처 회고: [핵심 결정 1개]

> **TL;DR**: [결정]을 했고 [결과]였다. 다시 한다면 [변경점].

## Context
- 프로젝트 배경, 규모, 제약 조건
- 결정 시점의 상황

## 고려한 선택지
### Option A: ...
장점 / 단점

### Option B: ...
장점 / 단점

## 우리의 선택과 이유
- 선택한 옵션과 핵심 근거

## 구현 후 6개월 — 무엇이 맞고 틀렸나
- 예상대로 된 것
- 예상과 달랐던 것
- 다시 한다면 바꿀 것

## 배운 원칙
- 일반화할 수 있는 설계 원칙 2~3개
```

## 블로그 글 작성 프로세스

### Step 1: 주제 파악
사용자에게 확인:
- 어떤 유형의 글인가? (심화/트러블슈팅/비교/튜토리얼/회고)
- 타겟 독자는? (주니어/미드레벨/시니어)
- 관련 코드베이스가 있는가?

### Step 2: 코드베이스 분석 (해당 시)
- 관련 코드 스캔 및 핵심 구현 추출
- git log에서 관련 커밋 히스토리 확인
- 테스트 코드에서 엣지 케이스 파악

### Step 3: 초안 작성
- 선택한 템플릿에 맞춰 구조 생성
- 코드 예시는 실제 프로젝트에서 추출 + 단순화
- Before/After 패턴으로 변화 시각화
- 벤치마크/측정 데이터 포함

### Step 4: SEO + LLM 최적화
- 타이틀: 60자 이내, 핵심 키워드 포함
- 메타 설명: 155자 이내 요약
- H2/H3 구조 최적화
- 내부 링크 (관련 글 연결)
- 코드 블록에 언어 태그 명시

### Step 5: 품질 검증

## 블로그 글 품질 체크리스트

### 필수 (MUST)
- [ ] TL;DR이 있는가 (바쁜 독자를 위한 3문장 요약)
- [ ] 실무 경험 기반인가 (이론만 나열하지 않았는가)
- [ ] 코드 예시가 있는가 (동작하는 코드)
- [ ] "왜?"가 설명되었는가 (기술 선택 근거)
- [ ] 한계점/트레이드오프를 솔직히 명시했는가
- [ ] References가 있는가 (공식 문서, 관련 자료)

### 권장 (SHOULD)
- [ ] Before/After 비교가 있는가
- [ ] 정량 데이터가 있는가 (벤치마크, 측정 수치)
- [ ] 다이어그램이 있는가 (아키텍처, 플로우)
- [ ] 관련 글 링크가 있는가 (시리즈 연결)
- [ ] 실행 환경이 명시되었는가

### 금지 (MUST NOT)
- [ ] ChatGPT/AI가 쓴 것 같은 뻔한 서론 (❌ "오늘날의 빠르게 변하는...")
- [ ] 근거 없는 추천 (❌ "~를 사용하는 것을 추천합니다")
- [ ] 튜토리얼 따라하기 수준의 내용
- [ ] 오래된 정보 (버전, API 변경사항 미반영)
- [ ] 지나치게 긴 코드 블록 (50줄 이상은 분할)

## 출력 형식

### 파일명 규칙
```
YYYY-MM-DD-[slug].md
예: 2026-03-05-rag-contextual-chunking-optimization.md
```

### Front Matter (공통)
```yaml
---
title: "[타이틀]"
date: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [tag1, tag2, tag3]
category: [deep-dive | troubleshooting | comparison | tutorial | retrospective]
description: "[메타 설명 155자 이내]"
toc: true
---
```

## 글감 발굴 가이드

코드베이스에서 블로그 글감을 자동 발굴할 때 찾을 것:

1. **git log에서 큰 리팩토링 커밋** → 회고/ADR 블로그
2. **복잡한 버그 수정 커밋** → 트러블슈팅 스토리
3. **새로운 라이브러리 도입 커밋** → 비교 분석 / 도입기
4. **성능 최적화 커밋** → 벤치마크 deep dive
5. **아키텍처 변경 (파일 구조 대규모 변경)** → 아키텍처 결정 기록
6. **테스트 추가 커밋** → 테스트 전략 / TDD 경험기
