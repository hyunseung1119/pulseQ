# PulseQ — Product Requirements Document (PRD)

## 1. 제품 개요

| 항목 | 내용 |
|------|------|
| 제품명 | PulseQ |
| 한 줄 설명 | 선착순 이벤트를 안전하고 공정하게 처리하는 실시간 대기열 SaaS API |
| 타겟 고객 | 이커머스, 티켓 플랫폼, 교육 기관 (수강신청), 부동산 (청약) |
| 핵심 가치 | 순간 트래픽 폭주에도 서버 다운 없이 공정한 순서로 처리 |
| 수익 모델 | API 호출 기반 과금 (Free: 1만 건/월, Pro: 100만 건/월) |

## 2. 문제 정의

### 고객의 Pain Point
1. **서버 다운**: 선착순 이벤트 시작 시 트래픽 폭주로 서비스 장애
2. **불공정**: 매크로/봇 사용자가 실제 사용자보다 먼저 확보
3. **구축 비용**: 자체 대기열 시스템 구축에 최소 3-6개월 + 전문 인력 필요
4. **일시적 수요**: 연 몇 회 이벤트를 위해 상시 인프라 유지 비효율

### 한국 시장 특수성
- 수강신청 (매 학기 수백만 동시 접속)
- 콘서트 티켓팅 (인터파크/멜론 — 수십만 동시 접속)
- 이커머스 타임딜 (쿠팡/11번가 — 플래시 세일)
- 부동산 청약 (순간 집중 트래픽)

## 3. 사용자 페르소나

### P1: 이커머스 CTO (김기술, 35세)
- 월 1-2회 한정판 세일 진행
- 세일 시작 시 서버 다운 경험 3회+
- 자체 구축 인력/시간 부족
- **니즈**: API 하나로 대기열 + 봇 차단 + 모니터링

### P2: 대학 IT 관리자 (박전산, 42세)
- 수강신청 기간 매 학기 서버 장애
- 학생 민원 폭주
- **니즈**: 공정한 순서 보장 + 실시간 현황 대시보드

### P3: 티켓 플랫폼 PM (이기획, 29세)
- 인기 공연 티켓 오픈 시 매크로 문제
- 실제 팬보다 리셀러가 선점
- **니즈**: 봇 탐지 + 차단 + 공정 배분

## 4. 핵심 기능 (MVP)

### Phase 1 — Foundation (2주)
| ID | 기능 | 우선순위 | 설명 |
|----|------|---------|------|
| F-001 | 테넌트 관리 | P0 | 회원가입, API 키 발급, 과금 플랜 설정 |
| F-002 | 이벤트 생성 | P0 | 대기열 이벤트 생성 (이름, 시작/종료 시간, 최대 인원, 처리 속도) |
| F-003 | JWT 인증 | P0 | API 키 기반 인증 + 테넌트 격리 |
| F-004 | 헬스체크 API | P0 | 서비스 상태 확인 엔드포인트 |

| ID | 기능 | 우선순위 | 설명 |

## 5. 비기능 요구사항

| 항목 | 목표 | 비고 |
|------|------|------|
| 응답 시간 (p99) | < 100ms | 대기열 입장/조회 API |
| 동시 처리 | 10,000 req/s (로컬) | k6 부하 테스트 검증 |
| 가용성 | 99.9% | 이벤트 진행 중 다운타임 0 |
| 데이터 정합성 | 순번 중복 0건 | Redis 분산 락 보장 |
| 봇 탐지율 | > 90% | Precision > 85%, Recall > 90% |
| Cold Start | < 3초 | 컨테이너 시작 후 요청 처리 가능 시간 |

## 6. 성공 지표 (KPI)

| 지표 | 목표 | 측정 방법 |
|------|------|----------|
| API p99 레이턴시 | < 100ms | Prometheus histogram |
| 동시 접속 처리량 | 10,000 req/s | k6 부하 테스트 |
| 봇 차단 정확도 | > 90% | 혼동 행렬 (Confusion Matrix) |
| 대기열 순번 정합성 | 중복 0건 | 통합 테스트 검증 |
| 테스트 커버리지 | > 80% | JaCoCo |

## 7. 제외 범위 (Out of Scope)

- 결제 시스템 직접 구현 (Webhook으로 고객사 결제 연동만)
- 모바일 앱 (Web Dashboard + REST API만)
- 다국어 지원 (한국어 우선)
- 실제 과금 결제 연동 (대시보드에서 사용량 표시만)

## 8. 리스크 및 대응

| 리스크 | 영향 | 대응 |
|--------|------|------|
| Redis 장애 시 대기열 유실 | Critical | Redis AOF 영속화 + Kafka 이벤트 소싱으로 복구 |
| 봇 탐지 오탐 (실사용자 차단) | High | 임계치 보수적 설정 + 수동 해제 API 제공 |
| Kafka Consumer 지연 | Medium | Consumer Group 스케일아웃 + lag 알림 |
| AWS 과금 폭탄 | High | 프리티어 범위 모니터링 + Billing Alert 설정 |
